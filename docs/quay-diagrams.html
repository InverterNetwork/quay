1. Actors and boundaries

Three actors run the loop. Anything that crosses a boundary becomes an artifact; in-actor work does not.

  Hermes -- "brief, ticket_snapshot,
submit-brief, escalate-human,
cancel" --> Quay
  Quay -- "preamble + brief
(.quay-prompt.md)" --> Worker
  Worker -- ".quay-blocked.md,
session_log, push, gh pr create" --> Quay
  Quay -- "polls PR, CI, reviews,

  Quay -- "post escalation,
poll replies" <--> Slack
  Worker -- "git push, gh pr create" --> GitHub
  Hermes -. "pulls task list" .-> Quay

Quay never initiates outbound to Hermes. Hermes polls; state is the signal. Worker has no Linear/Slack/orchestrator access.

4. Task state machine

One chokepoint validates every transition. Every respawn (orchestrator-driven or Quay-driven) routes back through queued; tick is the single spawn point.

stateDiagram-v2
  [*] --> queued : enqueue
(bootstrap synchronously)
  queued --> running : tick promotes
(capacity OK)
  running --> pr_open : dead, PR exists, progress
  running --> queued : crash / no_progress / stale /
wall_clock / malformed_signal /
ci_fail (later from pr-open)
  running --> awaiting_next_brief : blocker_written
  running --> queued : spawn_failed (rollback budget)
  running --> worktree_error : spawn_failures >= cap

  pr_open --> done : CI pass
  pr_open --> queued : CI fail (budget) /
review CHANGES_REQUESTED (non-budget) /
conflict (non-budget)
  pr_open --> merged : PR merged
  pr_open --> closed_unmerged : PR closed

  done --> merged : PR merged
  done --> closed_unmerged : PR closed
  done --> queued : review (non-budget) / conflict (non-budget)

  awaiting_next_brief --> claimed : claim
  claimed --> queued : submit-brief
  claimed --> waiting_human : escalate-human
  claimed --> awaiting_next_brief : release-claim /
claim timeout
  claimed --> orchestrator_loop : timeout cap reached

  waiting_human --> awaiting_next_brief : reply ingested

  pr_open --> non_budget_loop : non-budget cap (N+1)
  done --> non_budget_loop : non-budget cap (N+1)

  state "cancel_requested_at set" as cancel_intent
  cancel_intent --> cancelled : finalizer

  merged --> [*]
  closed_unmerged --> [*]
  cancelled --> [*]

Parked (manual recovery via quay cancel): worktree_error, orchestrator_loop, non_budget_loop. budget_exhausted is a flag, not a state — set when entering awaiting-next-brief with attempts_consumed >= retry_budget.

5. Tick cycle

One pass of quay tick. The supervisor lockfile serializes all side-effecting work and blocks quay cancel.

flowchart TD
  A[quay tick fires] --> B{acquire
tick.lock?}
  B -- no, held --> X[exit 0]
  B -- yes --> C[load active tasks]
  C --> D[for each task in own try/except]
  D --> E{cancel_requested_at
set?}
  E -- yes --> F[run cancel_finalizer]
  F --> N[next task]
  E -- no --> G{state}
  G -- queued --> Q[promote if capacity:
fetch, snapshot remote SHA + PR-exists,
SQL: spawned_at, attempts_consumed if budget,
state=running, write spawned event
then substrate spawn]
  G -- running --> R[dead-worker classifier
or staleness/wall-clock kill]
  G -- pr_open --> P1[poll PR/checks/mergeable]
  G -- done --> P2[poll PR/review/mergeable]
  G -- waiting_human --> S[Slack handler:
fence, nonce-recover, post, poll replies]
  G -- claimed --> CL{claim_age >
timeout?}
  CL -- yes --> CLA[auto-release; ++expirations;
orchestrator_loop if cap]
  CL -- no --> N
  G -- awaiting_next_brief --> N
  G -- terminal/parked --> N
  Q --> N
  R --> N
  P1 --> N
  P2 --> N
  S --> N
  CLA --> N
  N --> D
  D --> Z[release lock]
  Z --> Y[stdout: NDJSON,
one line per task touched]

Per-task error isolation: a failure in one task logs tick_error and continues. A tick never aborts a cycle because of one task. Every tick-owned mutating SQL also includes AND cancel_requested_at IS NULL as belt-and-suspenders.

6. Spawn / promotion lifecycle

Attempts have two phases. Scheduling and promotion are decoupled, which is what makes capacity, budget, and recovery work uniformly across initial and respawn paths.

7. Dead-worker classifier

Single classifier serves both the normal "dead worker" branch and the spawn-window tmux_session IS NULL recovery. Worker-alive precedes PR check; PR creation itself counts as progress.

flowchart TD
  Start[Worker observed dead] --> KI{kill_intent set?}
  KI -- wall_clock --> RW[exit_kind=killed_wall_clock
schedule wall_clock retry]
  KI -- stale --> RS[exit_kind=killed_stale
schedule stale retry]
  KI -- cancel --> RC[run cancel_finalizer]
  KI -- none --> Log[collect session log]
  Log --> Sig{.quay-blocked.md
present?}
  Sig -- valid --> Bl[ingest blocker artifact;
state → awaiting-next-brief;
set budget_exhausted if cap reached]
  Sig -- malformed --> Mf[persist malformed_signal;
schedule malformed_signal retry
state → queued]
  Sig -- absent --> Fetch[git fetch origin branch;
read remote_sha_at_exit;
gh pr view → pr_exists_at_exit]
  Fetch --> Prog{progress?
remote advanced
OR PR created this attempt}
  Prog -- yes, PR exists --> PrO[exit_kind=pr_opened
state → pr-open]
  Prog -- no, PR exists --> NoProg[exit_kind=no_progress
schedule crash retry
state → queued]
  Prog -- no, no PR --> Crash[exit_kind=crashed
schedule crash retry
state → queued]

"Progress" = remote advanced OR new PR opened during this attempt. Local-only commits do not count. spawn_failed only fires as the no-evidence default in the spawn-window recovery; it rolls back the budget increment. Evidence-found recovery preserves budget.

8. Claim & submit-brief flow

The orchestrator pull loop is a thin shell around task list + atomic task claim. Every claim mints a fresh UUID; every claim-scoped write is fenced by it.

sequenceDiagram
  autonumber
  participant Cron as cron
  participant Pull as pull script
  participant Quay as quay CLI
  participant DB as SQLite
  participant Hermes as Hermes (LLM)

  Cron->>Pull: every N min
  Pull->>Quay: task list --state awaiting-next-brief
  Quay->>DB: SELECT tasks WHERE state='awaiting-next-brief'
  Quay-->>Pull: JSON array
  loop for each task_id
    Pull->>Quay: task claim «id»
    Quay->>DB: UPDATE tasks SET state='claimed-by-orchestrator',
claim_id=new_uuid, claimed_at=now()
WHERE state='awaiting-next-brief'
AND cancel_requested_at IS NULL
    alt rowcount=1
      Quay-->>Pull: {task_id, claim_id}
      Pull->>Hermes: invoke --task-id «id»
      Hermes->>Quay: task get / artifact get
      Hermes->>Quay: submit-brief --claim-id «uuid» --reason blocker_resolved
OR escalate-human --claim-id «uuid»
      Quay->>DB: UPDATE WHERE state='claimed-by-orchestrator'
AND claim_id=? AND cancel_requested_at IS NULL
      alt rowcount=1
        DB-->>Quay: ok · clear claim_id · reset expirations
        Quay-->>Hermes: ok
      else mismatch
        Quay-->>Hermes: error: claim_lost / cancelled / wrong_state
      end
    else rowcount=0
      Quay-->>Pull: error (someone else got it)
    end
  end

  Note over Pull,DB: If Hermes crashes mid-work,
tick auto-releases after claim_timeout_seconds.
After max_claim_expirations consecutive timeouts
→ state=orchestrator_loop (parked).

9. Slack escalation sequence

Tick is the only Slack writer. The CLI persists artifact + state and returns; the actual API call lands within one tick interval. Crash recovery uses a per-escalation nonce.

sequenceDiagram
  autonumber
  participant Op as Hermes
  participant CLI as escalate-human
  participant DB as SQLite
  participant Tick as tick (waiting_human handler)
  participant Slack as Slack API

  Op->>CLI: escalate-human --claim-id ... --question-file q.md
  CLI->>DB: BEGIN
fence: state=claimed AND claim_id=? AND cancel NULL
++next_escalation_seq → seq
nonce = quay-esc-«short»-«seq»-«rand8»
INSERT slack_escalation_post artifact
(content_hash over body||seq||nonce ·
fence_ts/post_ts/recovered_ts NULL)
UPDATE tasks SET state='waiting_human', claim_id=NULL
COMMIT
  CLI-->>Op: {ok}

  Note over Tick,Slack: next tick — under tick.lock
  Tick->>DB: SELECT latest slack_escalation_post for task
  alt fence_ts NULL
    Tick->>Slack: conversations.replies (read latest ts)
    Slack-->>Tick: «fence_ts»
    Tick->>DB: UPDATE artifact SET slack_pre_post_fence_ts=«fence»
  end
  alt recovered_ts NULL
    Tick->>Slack: conversations.replies — search bot msgs containing nonce
    alt match
      Slack-->>Tick: bot_ts
      Tick->>DB: UPDATE artifact SET slack_recovered_post_ts=bot_ts
(also slack_post_ts if NULL)
    else no match
      Tick->>Slack: chat.postMessage
body = question + "\n\n_«nonce»_"
      Slack-->>Tick: returned_ts
      Tick->>DB: UPDATE artifact SET slack_post_ts=ts,
slack_recovered_post_ts=ts
    end
  end
  Tick->>Slack: poll replies WHERE ts > (recovered_ts ?? fence_ts)
  alt non-bot reply found
    Slack-->>Tick: reply
    Tick->>DB: INSERT slack_reply artifact (content_hash) ·
state → awaiting-next-brief
  else nothing yet
    Tick-->>Tick: skip · retry next tick
  end

If tick crashes after the Slack post but before persisting the ts, the next tick finds the bot message by nonce — never duplicate posts. slack_recovered_post_ts is set exactly once and never updated.

10. Cancel finalizer

Single canonical sequence used by both quay cancel (synchronous) and tick recovery. Idempotent on already-cancelled.

flowchart TD
  A["quay cancel «id»"] --> B["acquire tick.lock
(blocks if tick in flight)"]
  B --> C["BEGIN txn
cancel_requested_at = now,
cancel_close_pr, cancel_keep_worktree;
if state=running:
  attempts.kill_intent='cancel'
COMMIT"]
  C --> D[run finalizer synchronously]
  D --> E["Step 1: ensure no live worker
tmux kill canonical session
(idempotent)"]
  E --> F[Step 2: best-effort collect session log]
  F --> G["Step 3: cleanup per matrix
--close-pr → gh pr close;
delete local branch;
remote branch delete unless PR open;
remove worktree (or retain if --keep-worktree)"]
  G --> H["Step 4: BEGIN txn
attempts.exit_kind=killed_cancel
(only if ended_at NULL);
clear kill_intent;
state='cancelled';
clear claim_id, claimed_at;
write cancelled event
COMMIT"]
  H --> I[Step 5: reset transient counters
tick_error, claim_expirations,
spawn_failures]
  I --> J[release tick.lock]

  X[crash between any step] -.-> Y[next tick:
cancel_requested_at IS NOT NULL
→ re-enter at step 1]
  Y -.-> E

  classDef bad fill:#ffeaea,stroke:#aa3333;
  class X bad

Top-of-tick cancel check applies to every non-terminal state — even done, waiting_human, parked. Re-running on cancelled is a no-op success. cancel_requested_at is never cleared (forensic record).

11. Retry / budget decision

Triggers schedule; promotion consumes. Two distinct accounting tracks: budget vs. non-budget.

flowchart TD
  T[trigger detected]
  T --> Kind{trigger kind}

  Kind -- "ci_fail / crash / stale /
wall_clock / malformed_signal /
blocker_resolved / initial" --> Budg{attempts_consumed
< retry_budget?}
  Budg -- yes --> Sched["INSERT attempts
(spawned_at NULL,
consumed_budget=1);
state=queued"]
  Budg -- no --> Exh[persist last_failure artifact;
budget_exhausted=true;
state=awaiting-next-brief]

  Kind -- "review / conflict
(non-budget)" --> Inc[++ non_budget_respawns_consumed
in same txn]
  Inc --> Cap{post-increment
> cap?}
  Cap -- yes --> Park["no schedule;
state=non_budget_loop
(parked)"]
  Cap -- no --> SchedNB["INSERT attempts
(consumed_budget=0);
record dedupe key
(last_review_id_acted_on /
last_conflict_observation);
state=queued"]

  Kind -- "advice_answered
(after Slack reply)" --> AA["INSERT attempts
(consumed_budget=0);
state=queued
(NOT counted in non-budget cap)"]

  Sched --> Promote[tick promotes when capacity:
++attempts_consumed
iff consumed_budget=1]
  SchedNB --> Promote
  AA --> Promote

Budget-consuming spawn reasons: initial, blocker_resolved, ci_fail, crash, stale, wall_clock, malformed_signal. Non-consuming: advice_answered, review, conflict. Spawn-failure no-evidence path rolls back the budget increment from promotion.

14. Enqueue bootstrap
